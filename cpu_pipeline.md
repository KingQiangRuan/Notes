1.  CPU流水线
  1) IF (取指)
  2) ID (译码)
  3) EX (执行)
  4) MEM (访存)
  5) WB  (写回)

2. CPU 流实现冒险
  1) 结构冒险
       硬件冲突，比如 IF, MEM 都需要访问内存读取数据，而内存只有一个地址译码器，在一个周期内只能读取条数据
    
   解决方案：
    把内存分为两部分，存放指令和存放数据的内存，有各自的地址译码器

  指令和数据存储分开：Harvard Architecture
  指令和数据放在一起：冯诺依曼结构
  现代CPU采用高速cache解决，综合两种结构

 2) 数据冒险
   指令有相互依赖
例：
   int a = 10; 
   int b = a; // 语句2
   int c =  b; // 语句3

  语句3 的执行需要等语句2执行结束。 而一条语句的被拆分成几个指令，那么， 指令执行有相互依赖
   指令2 需要WB 后，指令3 才能EX， 如果不等WB，那么执行得到的结果就是错误的。

解决方案：
 1. NOP操作，在当前时钟周期啥也不做，等待依赖的数据完成在执行，即流水线冒泡（pipeline Bubble）.  简单单效率低
 2. 在指令2 执行的结果数据直接给到指令的3的 ALU，下一条指令就不需要插入NOP。
   即操作前推\转发(Operand Forward) 和NOP一起使用
   
 3) 控制冒险
  指令转移，会改变指令的流向，和pipeline会有冲突，pipeline 希望的是一次进入流水线
  
  i) 有条件转移
    在EX阶段才能确定，这样引入分支预测
  ii)  无条件转移
   确定发生，在IF阶段就可以确定，更早地把结果反馈到pipeline，对应bypass 电路

3. CPU 分支预测
  pipeline 的上一个指令没执行完，但是下一个指令依赖这个结果，pipeline 不能停顿，做出预测

  预测 是跳转或顺序执行 两种情况

i) 静态预测
  每次都猜同一个结果，有50%对的概率，效率低
ii) 动态预测
  根据之前的选择和正确率来预测

a) 预测成功
   尽快找到分支目标指令地址
 
b) 预测失败
  作废 IF、ID,  恢复现场，从另一个分支路径重新取指令。
